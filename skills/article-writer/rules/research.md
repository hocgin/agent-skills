# 信息调研规则

> **目标**：基于brief需求，进行系统化的信息搜索和调研，为写作提供可靠的数据支持

## 功能说明

- 自动检测调研类型（文档网站 / PDF / 常规搜索）
- 支持文档网站自动爬取和建立知识库
- 支持PDF文档提取
- 支持常规Web搜索调研
- 自动整理结构化调研报告

---

## 输入参数

### 方式1：指定文档URL（推荐）

```
/research https://vuejs.org/guide/
```

AI会自动检测并爬取整个文档网站。

### 方式2：指定brief文件

```
/research _briefs/001-brief.md
```

### 方式3：无参数（自动推断）

```
/research
```

AI会查找最新的brief文件。

---

## 处理流程

### 1️⃣ 检测调研类型

#### URL自动检测

如果用户提供了URL，AI会检查URL特征：

| 特征 | 类型 | 处理方式 |
|------|------|----------|
| 包含 `docs.`、`/docs/`、`/guide/`、`/api/` | 文档网站 | 模式A：文档爬取 |
| 文件扩展名为 `.pdf` | PDF文档 | 模式B：PDF提取 |
| 其他URL | 网页 | 模式C：常规搜索 |

#### 无URL时

- 如果有brief参数 → 模式D：Brief调研
- 无参数 → 模式D：自动查找最新brief

---

### 2️⃣ 模式A：文档爬取（自动化）

**适用场景**：官方技术文档、API文档、教程网站

**执行步骤**：

**步骤1：提取基础信息**
```
输入: https://vuejs.org/guide/introduction
提取: https://vuejs.org/guide/
项目名: vuejs-docs
```

**步骤2：执行爬取**

AI应提示用户：
```
🚀 检测到文档网站，启动自动爬取系统...

📚 正在爬取: Vue.js 官方文档
📍 URL: https://vuejs.org/guide/
⚙️  配置: 最大200页，并发5个

预计时间: 10-15分钟
```

**步骤3：使用WebFetch工具爬取**

对文档网站的每个页面：
- 使用 WebFetch 获取页面内容
- 提取主要内容（去除导航、侧边栏等）
- 转换为Markdown格式
- 保存到 `_knowledge_base/raw/[项目名]/`

**步骤4：生成索引**

创建 `_knowledge_base/indexed/[项目名]-index.md`，包含：
- 文档结构（目录树）
- 各部分内容摘要
- 关键概念索引
- 代码示例链接

**步骤5：输出结果**

```
✅ 文档爬取完成！

📊 爬取成果:
  - 总页数: 186页
  - 分类: 入门指南/API参考/示例代码等5个
  - 存储位置: _knowledge_base/raw/vuejs-docs/
  - 知识库索引: _knowledge_base/indexed/vuejs-docs-index.md

💡 后续写作时，AI会自动搜索这个知识库
```

---

### 3️⃣ 模式B：PDF提取（自动化）

**适用场景**：产品手册、研究报告、论文PDF

**执行步骤**：

**步骤1：检测PDF来源**
- URL：先下载到临时目录
- 本地路径：直接使用

**步骤2：读取PDF内容**

使用Read工具读取PDF文件（支持分页读取）。

**步骤3：转换为Markdown**

- 提取文本内容
- 识别代码块
- 保留标题结构
- 保存到 `_knowledge_base/raw/[项目名].md`

---

### 4️⃣ 模式C：常规搜索调研

**适用场景**：需要多来源信息、最新讨论、用户评价

**搜索策略**：

#### A. 官方信息（P0优先级）

```
使用 WebSearch 搜索:
- "[主题] official documentation"
- "[主题] changelog 2025"
- "[主题] vs [对比对象] comparison"

使用 WebFetch 获取:
- 官方文档
- 官方博客
- GitHub README/Release Notes
```

#### B. 用户评价（P1优先级）

```
使用 WebSearch 搜索:
- "[主题] review reddit"
- "[主题] user experience"
- "[主题] 评测"

关注来源:
- Reddit、Hacker News
- Twitter/X、即刻、知乎
```

#### C. 技术评测（P1优先级）

```
使用 WebSearch 搜索:
- "[主题] hands-on review"
- "[主题] benchmark test"
- "[主题] 深度评测"

关注来源:
- 技术博客、Medium文章
- YouTube评测（文字描述）
```

---

### 5️⃣ 模式D：Brief调研

**适用场景**：基于brief需求进行传统调研

**执行步骤**：

**步骤1：读取Brief**

提取关键信息：
- 主题关键词
- 对比对象
- 数据要求
- 内容定位（评测/教程/观点）

**步骤2：确认调研范围**

```
💡 建议调研方向:
1. [主题1] 官方文档（功能、定价、限制）
2. [主题2] 官方对比资料
3. 用户评价和社区讨论
4. 技术评测文章（最近3个月）

❓ 请确认:
- 是否需要补充其他方向?
- 调研深度: [快速概览 / 深度调研]?
- 时效性要求: [不限 / 近1个月 / 近3个月]?
```

**步骤3：执行搜索**

使用模式C的搜索策略进行调研。

## 质量标准

### 信息可靠性
- ✅ 优先使用官方来源
- ✅ 用户评价需标注来源和时间
- ✅ 技术评测需检查作者权威性
- ❌ 避免使用未验证的二手信息

### 完整性检查
- [ ] Brief中的必须调研项(P0)是否全部覆盖?
- [ ] 是否有足够的对比数据?
- [ ] 是否有真实用户评价?
- [ ] 来源是否多样化?

### 时效性检查
- [ ] 信息是否为最新版本?
- [ ] 用户评价是否在有效期内(如近3个月)?
- [ ] 是否标注了信息获取时间?

---

## 注意事项

### 1. 搜索策略
- **广度优先**: 先快速浏览多个来源，再深入重点内容
- **来源多样化**: 不要只依赖单一平台或观点
- **时效性**: 技术产品更新快，优先使用最新信息

### 2. 信息筛选
- **官方 > 技术博客 > 用户评价 > 营销内容**
- 识别并过滤营销性质的"软文"
- 注意用户评价的代表性(避免极端案例)

### 3. 数据引用
- **必须标注来源URL**
- 原文引用使用引号或代码块
- 如有数据冲突，记录多个来源并标注

## 常见问题

### Q1: 如果找不到对比数据怎么办?

**A**:
1. 调整搜索关键词(尝试英文/中文)
2. 扩大搜索范围(GitHub Issues/Discord讨论)
3. 在调研报告中明确标注"未找到公开数据"
4. 建议在写作时标注"待验证"

### Q2: 用户评价相互矛盾怎么办?

**A**:
1. 记录不同观点及其来源
2. 分析矛盾原因(使用场景不同?版本不同?)
3. 在调研报告中呈现多样化观点
4. 标注需要进一步验证

### Q3: 调研时间应该控制在多久?

**A**:
- **快速概览**: 30-60分钟(P0信息)
- **标准调研**: 2-3小时(P0+P1信息)
- **深度调研**: 半天(P0+P1+P2信息 + 深度分析)

根据brief的复杂度调整。

## 工具使用说明

**注意**：以下工具的可用性取决于您使用的 AI 环境（Claude Code、Cursor 等）。

### WebSearch
用于搜索最新信息和讨论。
- **用途**：获取最新资讯、用户评价、技术讨论

### WebFetch / mcp__web_reader__webReader
用于获取具体页面内容。
- **用途**：读取网页完整内容，支持 PDF 文档
- **优先级**：WebFetch（原生） > mcp__web_reader__webReader（MCP）

### Read
用于读取本地文件和 PDF。
- **用途**：读取本地 Markdown、PDF 文档

### Write
用于保存调研报告和知识库文件。
- **用途**：将调研结果保存到 `_knowledge_base/` 目录

### Bash（可选）
用于执行爬虫脚本（如果配置了自定义脚本）。
- **用途**：批量爬取文档网站
