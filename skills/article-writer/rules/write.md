# 快速写作规则

> **目标**：基于调研结果和brief，AI自动生成高质量初稿，目标AI检测率 < 25%

## 核心原则

- **AI生成初稿**：100%由AI生成，无需人工参与写作
- **基于调研数据**：引用调研报告中的具体数据、事实、案例
- **口语化表达**：降低AI味，使用自然流畅的表达
- **支持迭代修改**：最多3轮迭代，逐步优化内容

---

## 前置条件

开始写作前，确保已完成：

- ✅ **Brief定义**：有明确的需求文档（`_briefs/XXX-brief.md`）
- ✅ **信息调研**：有调研报告（`_knowledge_base/XXX-调研报告.md`）
- ✅ **选题讨论**：有文章结构（可选，可从brief中提取）

### 2️⃣ 生成初稿

#### 生成原则

**1. 严格遵循Brief要求**
- 按照brief中的字数要求生成
- 覆盖所有P0优先级要点
- 符合目标读者的阅读习惯

**2. 引用调研数据**
- 引用具体数字、事实、用户评价
- 标注所有数据来源（`[来源: ...]`）
- 优先使用官方数据和用户真实评价

**3. 口语化表达（降低AI味）**

| AI味表达 | 改为 |
|---------|------|
| "经过深入分析，我们可以得出结论" | "测试下来，我的感受是" |
| "该工具在性能方面表现优异" | "速度确实快，这点没得说" |
| "综上所述" | "说实话" / "总的来说" |
| "一方面...另一方面..." | "有个好消息，也有个坏消息" |
| "值得注意的是" | 删除，直接说重点 |

**4. 避免AI套话**

❌ **避免使用**：
- "综上所述"、"总而言之"、"可以看出"
- "值得注意的是"、"需要指出的是"
- "不仅...而且..."、"既...又..."
- 过度对称的结构（优点123，缺点123）
- 空洞的形容词堆砌（"非常强大"、"极其优秀"）

✅ **推荐使用**：
- 口语化连接词（"说实话"、"不过话说回来"、"嗯..."）
- 具体数据替代抽象描述
- 个人观点和判断
- 反问句和语气词

**5. 增加真实感**

- 加入具体场景和数字
- 引用真实用户评价
- 提及踩坑经历或注意事项
- 使用"我发现"、"说实话"、"个人建议"等表达

```markdown
**结论**：对于需要深度理解代码的场景，Claude Code明显更强。
```

### 4️⃣ 迭代修改（最多3轮）

**用户反馈后，AI的处理流程**：

**步骤1：理解修改需求**
- 提取具体的修改点
- 确认修改范围（某一段 vs 全文）

**步骤2：执行修改**
- 使用Edit工具修改draft.md
- 只修改指定的部分，不动其他内容

**步骤3：展示修改结果**

```markdown

✅ 已修改: [修改点2]

- 总字数: [XXXX]字
```

## 降低AI味的具体技巧

### 1. 开头避免套话

❌ **错误**：
```markdown
随着人工智能技术的快速发展，AI编程工具已经成为开发者的必备工具。本文将对比Claude Code和Cursor两款主流工具...
```

✅ **正确**：
```markdown
说起AI编程工具，我算是重度用户了。过去三个月，Claude Code和Cursor轮换着用，基本每天都离不开。
```

### 2. 使用具体数据

❌ **错误**：
```markdown
Claude Code的理解能力很强
```

✅ **正确**：
```markdown
Claude Code看了不到1分钟，直接给出了"这个函数是处理用户登录状态的吧?"的提示
```

### 3. 加入个人观点

❌ **错误**：
```markdown
两款工具各有优势，用户可根据需求选择
```

✅ **正确**：
```markdown
我的结论是：新项目用Claude Code，老项目维护用Cursor。这不是最"科学"的选择，但符合我的使用习惯
```

### 4. 使用口语化连接

❌ **错误**：
```markdown
然而，值得注意的是，该工具在性能方面表现优异
```

✅ **正确**：
```markdown
说实话，速度确实快，这点没得说
```

### 5. 避免机械对称

❌ **错误**：
```markdown
优点:
1. 速度快
2. 功能全
3. 价格低

缺点:
1. 学习成本高
2. 偶尔不稳定
3. 文档不全
```

✅ **正确**：
```markdown
Cursor最打动我的是速度，真的快。但用了一段时间发现，快也有快的代价——有时候太激进了，会改一些不该改的代码。这不算严重缺点，养成review的习惯就好。
```

## 常见问题

### Q1: 初稿AI检测率太高怎么办?

**A**:
1. 执行`/review style`进行风格审校
2. 补充更多真实数据和案例
3. 加入更多个人观点和表达

**A**:
1. 补充具体案例和使用场景
2. 增加数据对比和分析
3. 扩展某个章节的细节描述

### Q3: 内容太枯燥怎么办?

**A**:
1. 加入真实使用场景
2. 引用用户真实评价
3. 加入踩坑经历和注意事项
4. 使用更口语化的表达
